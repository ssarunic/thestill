## Executive Summary

For a foreign-language podcast — e.g. the Croatian feed **Explora**
(`/podcasts/explora/episodes/explora-16-06-2026`) — the transcript is
Croatian but the summary is currently produced in **English**. This spec makes
the summary follow the podcast's language, and adds a UI toggle so a reader can
flip between the **original language** and their **default language**.

The key reframe: everything upstream of the summary *already runs in the
original language*. Podcasts carry an ISO `language` captured from the RSS
`<language>` tag ([podcast.py:315](../thestill/models/podcast.py#L315)), and
both transcription and transcript-cleaning already pass `podcast.language`
through the pipeline. The **only** step that forces English is the summarizer's
system prompt, which hardcodes *"Use British English"*
([post_processor.py:69](../thestill/core/post_processor.py#L69)). So today's
English summary of a Croatian podcast is the *extra* translation step — the
original-language summary is the more natural output.

That flips the cost model: **original-language is the base artifact; the user's
default language is an on-demand, cached translation** — not a second equal
generation. For the common case (English feed, English default) podcast
language == default language, so nothing changes: one summary, no toggle.

---

## Motivation

- Readers of a Croatian/German/Spanish podcast get an English summary of
  content they consume in the original language — a jarring mismatch, and a
  lossy one (idiom, names, tone).
- The infrastructure to do better already exists: podcast `language` is
  captured end-to-end and the cleaned transcript fed to the summarizer is
  already in the source language. Only the final prompt discards it.
- A toggle (rather than a hard switch) respects the case where a reader
  genuinely wants the summary in their own language — common for language
  learners and for skimming foreign feeds.

## Design

### Two summary languages, one base + one derived

Define two roles per episode summary:

- **Original** — generated by the summarizer in `podcast.language`. This is the
  canonical artifact and is always produced by the pipeline.
- **Translated** — the summary rendered in the reader's **default language**.
  Produced **lazily** the first time a reader requests it and cached to storage.

When `podcast.language == default_language`, the two collapse: there is only
one summary and **no toggle is shown**. This is the overwhelmingly common path
and must incur zero extra cost.

**Why lazy translation over pre-generating both:** pre-generating pays 2× LLM
cost on every foreign-language episode whether or not anyone reads the second
language. Lazy translation pays only when a reader actually flips the toggle,
then caches — so a never-toggled episode costs exactly the same as today.
The trade-off is a one-time latency + cost on first toggle (mitigated by
caching and an optimistic loading state). Accepted.

### Storage: language-keyed summary artifacts

Today one file per episode via `summary_path`. Extend to a small set of files
keyed by language code, without breaking the existing single-path contract.

- **Chosen approach:** keep `episodes.summary_path` as the **original-language**
  summary (back-compatible — existing rows already hold the canonical summary),
  and store translations as **sibling files** discovered by suffix:
  `…_summary.md` (original) alongside `…_summary.<lang>.md` (translation).
  Resolution is a filename convention; no schema change to `episodes`.
- `get_summary` gains an optional `language` argument
  ([podcast_service.py:812](../thestill/services/podcast_service.py#L812)):
  - `language is None` or `== podcast.language` → return the original.
  - `language == some other code` → return `…_summary.<lang>.md` if present,
    else `None` (caller triggers translation).

> **FM-6 guard:** the storage/read path must behave identically on SQLite and
> Postgres. Because we lean on FileStorage + a filename convention (not a DB
> column), there is nothing DB-specific to keep in lockstep here — but any
> future move to a `summary_translations` table must land in **both**
> repositories in the same change. Called out explicitly per spec #42 FM-6.

### Summarizer: language-aware prompt

- Add `language: str` to `EpisodeMetadata`
  ([post_processor.py:36](../thestill/core/post_processor.py#L36)) or to
  `summarize()` ([:280](../thestill/core/post_processor.py#L280)).
- Make the tone instruction conditional: replace the hardcoded *"Use British
  English"* ([:69](../thestill/core/post_processor.py#L69)) with *"Write the
  entire summary in {language_name}."* when `language != "en"`, keeping British
  English as the `en` default.
- **Section scaffolding** ("The Gist", "Timeline", "Key Takeaways", …) is
  currently English prose in the prompt. Decision needed (see Open Questions):
  instruct the model to localize the headers, or keep structural headers in
  English while the body is localized. Recommend **localize headers too** — a
  half-translated summary reads worse than a fully localized one.

### Translation path (lazy)

When a translated summary is requested and absent:

1. Load the original-language summary markdown.
2. Call the LLM with a translation prompt: *"Translate this Markdown summary to
   {target_language}. Preserve all Markdown structure and any `[[…]]` /
   citation link syntax exactly."*
3. Persist as `…_summary.<lang>.md` via FileStorage.
4. Re-resolve citations for the translated text (see Citations below).

This reuses the existing LLM provider abstraction — no new provider wiring.

### API + client

- `GET /{podcast_slug}/episodes/{episode_slug}/summary`
  ([api_podcasts.py:392](../thestill/web/routes/api_podcasts.py#L392)) gains an
  optional `?lang=<code>` query param.
  - Response envelope gains `language` (the language of the returned `content`)
    and `podcast_language` (so the client can decide whether to show a toggle)
    and `available_languages` (which variants are cached / offerable).
  - First request for an uncached translation triggers generation server-side
    (may be slower); subsequent requests hit the cache.
- Client `getEpisodeSummary` ([client.ts:198](../thestill/web/frontend/src/api/client.ts#L198))
  and hook `useEpisodeSummary` ([useApi.ts:229](../thestill/web/frontend/src/hooks/useApi.ts#L229))
  gain a `lang` argument, keyed into the react-query cache key so each language
  variant caches independently.

### Frontend toggle

- Shown **only when** `podcast_language != default_language`
  ([EpisodeReader.tsx](../thestill/web/frontend/src/components/EpisodeReader.tsx)).
- Two-state control near the summary tab: **"HR (original)"** ⇄ **"EN"** (labels
  derived from the two language codes).
- Selected language reflected in a URL param (e.g. `?lang=`) so Back/refresh and
  the list-filter-restoration convention are preserved (per the list-pages
  filter+scroll rule). Loading state while a translation is generated on first
  toggle.

### User default language

No `User.language` field exists today; `region` is a country code, not a
language.

- **v1 (recommended):** derive default language from the **browser locale**
  (`navigator.language` → primary subtag). Zero schema change; good enough to
  decide "translate to what?" for the toggle.
- **v2 (optional):** add `User.language` ([user.py:58](../thestill/models/user.py#L58)
  neighborhood) + a Settings selector mirroring the existing Region dropdown
  ([Settings.tsx](../thestill/web/frontend/src/pages/Settings.tsx)). Deferred —
  not required to ship the toggle.

## Citations Caveat

Summary citations are resolved against the summary **text**
([podcast_service.py:848](../thestill/services/podcast_service.py#L848)
`get_summary_citations_for_episode`, spec #54). A translated summary needs its
citation anchors to survive translation:

- The translation prompt must preserve the citation link syntax verbatim.
- Citation resolution runs against the translated text to produce a
  translation-specific citation set (anchor offsets differ from the original).
- If preserving anchors through translation proves unreliable, fall back to
  serving the translated summary **without** inline citations in v1 and note
  the gap — a readable translated summary beats a citation-perfect one nobody
  can read.

## Scope

### In scope

- Language-aware summarizer prompt (original-language generation).
- Lazy, cached translation to the reader's default language.
- Language-keyed summary storage (filename convention).
- `?lang=` API + client + react-query cache keying.
- Frontend toggle, shown only for foreign-language podcasts.
- Browser-locale-derived default language (v1).

### Out of scope (v1)

- `User.language` persisted setting + Settings UI (v2).
- Re-summarizing / backfilling the entire existing corpus into original
  language — applies going forward; a targeted backfill is a follow-up.
- Localizing the rest of the app UI (i18n of chrome/navigation).
- Translating transcripts or briefings (summary only).
- Per-episode language override (episodes inherit `podcast.language`).

## Build Sequence

1. **Summarizer**: thread `language` into `EpisodeMetadata` / `summarize()`;
   make prompt tone + headers conditional. Unit-test that a non-`en` language
   yields non-English output and `en` is unchanged.
2. **Storage + read**: `…_summary.<lang>.md` convention; `get_summary(…, language=)`
   resolution; keep `summary_path` = original.
3. **Translation service**: lazy translate-and-cache path reusing the LLM
   provider; citation re-resolution against translated text.
4. **API + client**: `?lang=` param, expanded response envelope, hook/client
   `lang` argument + cache key.
5. **Frontend toggle**: conditional control in `EpisodeReader`, URL-param state,
   loading state; browser-locale default.
6. **Pipeline**: ensure the `summarize` task handler
   ([task_handlers.py:659](../thestill/core/task_handlers.py#L659)) passes
   `podcast.language` so new summaries are generated in the original language.

Each step is independently testable and shippable behind the "only show toggle
when languages differ" gate.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Doubling summary cost | Lazy translation — only pay on first toggle, then cache. Common (English) path unchanged. |
| Citation anchors break under translation | Preserve link syntax in the prompt; re-resolve against translated text; v1 fallback to no-citation translated summary. |
| Eval rubric (#53) assumes English | Scope evals to `en` for now; note that the rubric needs a language-agnostic pass (follow-up). |
| SQLite/Postgres drift (#42 FM-6) | Storage is FileStorage + filename convention (DB-agnostic); any future table lands in both repos together. |
| Backfilled corpus still English | Applies going forward; targeted re-summarize is a separate follow-up, not a blocker. |

## Open Questions

1. **Localize section headers** ("The Gist" → Croatian) or keep structural
   headers English with a localized body? *Recommendation: localize fully.*
2. **Default-language source** — browser locale (v1, zero schema) vs. a real
   `User.language` setting (v2)? *Recommendation: browser locale for v1.*
3. **First-toggle UX** — generate translation synchronously on the request
   (simpler, slower first paint) or enqueue + poll? *Recommendation:
   synchronous for v1; revisit if p95 latency hurts.*
